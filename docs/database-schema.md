# Canonical Database Schema

All backend variants (dotnet, spring, nestjs) within the same variant level (full or minimal) **must produce identical database schemas**. This document defines the canonical schema that all implementations must match.

## Conventions

- **Column naming**: `snake_case` (PostgreSQL convention)
- **Primary keys**: `BIGINT GENERATED BY DEFAULT AS IDENTITY`
- **Timestamps**: `TIMESTAMPTZ` (timestamp with time zone)
- **Booleans**: `BOOLEAN` with explicit `DEFAULT` values
- **Audit fields**: `created_by` / `updated_by` are `VARCHAR(100)` strings (not foreign keys)
- **Roles**: Lowercase strings (`'admin'`, `'user'`)
- **Enums**: Stored as `VARCHAR` strings (not database enums)

## Minimal Variant (3 tables)

The minimal variant uses **external authentication (SSO/shared-secret JWT)**. There are no `users`, `departments`, or `refresh_tokens` tables.

### notifications

Same as full variant except:

| Column  | Change                                               |
|---------|------------------------------------------------------|
| user_id | `VARCHAR(100) NOT NULL` — string from JWT claims, **no FK** |

### uploaded_files

Identical to full variant.

### audit_logs

Identical to full variant.

CREATE INDEX ON audit_logs(entity_id);

## Seed Data (full variant only)

All full variants seed the following data on first startup:

```sql
-- Department
INSERT INTO departments (code, name, description, is_active, created_by)
VALUES ('GEN', 'General', 'Default department', true, 'system');

-- Admin user (password: Admin@123)
INSERT INTO users (username, email, password_hash, first_name, last_name, role, department_id, is_active, created_by)
VALUES ('admin', 'admin@apptemplate.com', '<bcrypt hash>', 'Admin', 'User', 'admin', <GEN id>, true, 'system');

-- Sample user (password: User@123)
INSERT INTO users (username, email, password_hash, first_name, last_name, role, department_id, is_active, created_by)
VALUES ('johndoe', 'user@apptemplate.com', '<bcrypt hash>', 'John', 'Doe', 'user', <GEN id>, true, 'system');
```

Minimal variants have **no seed data** (external auth, no users table).

## Implementation Notes

### .NET (EF Core)
- Snake-case naming via `EFCore.NamingConventions` (`UseSnakeCaseNamingConvention()`)
- Column types/lengths configured in `OnModelCreating()` via Fluent API
- Migrations generated via `dotnet ef migrations add`

### Spring Boot (Flyway)
- Schema defined in `V1__initial_schema.sql` (hand-written SQL)
- Seed data in `V2__seed_data.sql`
- JPA entities must match migration exactly (`@Column` annotations)
- `ddl-auto=validate` ensures entity/schema alignment

### NestJS (TypeORM)
- Schema defined in migration file (`1700000000000-InitialSchema.ts`)
- Entity decorators (`@Column`, `@PrimaryGeneratedColumn`) must match migration
- `synchronize: false` in production; migrations are authoritative
